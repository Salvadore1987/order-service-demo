# ADR-0004: Компенсация платежа через BPMN compensation event и retry-сабпроцесс оплаты

## Статус

Принято

## Контекст

Оплата и доставка — шаги с побочными эффектами. Если деньги списаны,
а доставка не удалась, заказ нельзя просто завершить с ошибкой — нужно
вернуть платёж (saga-семантика: компенсирующее действие вместо
распределённой транзакции). Кроме того, списание может временно
не проходить — нужны повторные попытки без падения процесса.

## Решение

### Компенсация (refund)

- На call activity **Charge Payment** в `order-process` навешан
  **boundary compensation event**;
- к нему привязана задача **Refund Payment**
  (`isForCompensation = true`, топик `order.payment.refund`,
  обработчик `OrderPaymentRefundTaskHandler`);
- при сбое доставки (`OrderDeliverTaskHandler` бросает RuntimeException)
  движок запускает компенсацию — уже выполненное списание откатывается
  возвратом платежа.

### Retry оплаты

Оплата вынесена в отдельный сабпроцесс `charge-payment-subprocess`
(call activity):

1. **Charge Payment** (`order.payment.charge`) — попытка списания;
   бизнес-неудача (`PaymentChargeException`, code 30000) завершает задачу
   с `isPaymentSuccess = false` (не technical error!);
2. exclusive gateway по `${isPaymentSuccess}`: успех → выход из сабпроцесса;
3. неуспех → **Check Status** (`order.payment.status`) — проверка статуса,
   инкремент `retryCount`;
4. снова gateway: успех → выход; неуспех → **timer PT5S** → возврат к шагу 1;
5. защита от бесконечного цикла: при `retryCount > 5` воркер списания
   возвращает техническую ошибку (`context.error`) — инцидент в движке.

## Последствия

**Плюсы:**

- Saga-компенсация средствами BPMN: откат платежа декларативен, виден на
  схеме, не требует самописного кода отката в сервисе.
- Retry с паузой 5с устойчив к временным сбоям платёжного провайдера;
  состояние ретраев персистентно в движке (переживает рестарт).
- Разделение бизнес-неудачи (ветка процесса) и технической ошибки
  (инцидент) — операционно прозрачно.

**Минусы:**

- Refund вызывается компенсацией только для уже **завершённых** задач
  Charge Payment — порядок и условия компенсации нужно аккуратно
  тестировать (покрыто `OrderProcessIntegrationTest.compensationFlow`).
- Лимит ретраев (5) и интервал (PT5S) зашиты в код/схему — при изменении
  требований нужна правка BPMN.
- Воркеры должны быть идемпотентны: задача может быть доставлена повторно.
